AI Infra 与大模型趋势雷达: 2026-06-20AI Infra & LLM Trend Radar: 2026-06-20

这是一份面向个人站点的 AI Infra / 大模型技术雷达快照,统计时间为 2026-06-20 22:00 UTC。信号来自 GitHub API、GitHub release、官方文档、技术博客和公开新闻报道。热度排序不是投资建议,而是按新鲜度、开源活跃度、生产影响、对推理成本/延迟的影响、是否改变开发与运维流程综合判断。

一句话结论: AI Infra 的中心正在从“把模型跑起来”转向“让 agent 便宜、稳定、可审计地连续运行”。当下最强信号不是单个模型发布,而是推理引擎、KV cache、低精度硬件、工具协议和代码平台一起被 agent 工作负载重写。

4. 技术趋势分析: 历史、当下、未来

历史: 2023-2025 是“推理引擎化”和“本地化”的形成期

2023 年前后的核心问题是如何把 Transformer decoder 经济地服务起来: continuous batching、PagedAttention、FlashAttention、量化、本地 GGML/llama.cpp、Ollama 的开发体验,共同解决“能跑、跑得起、开发者愿意用”。证据是今天仍然高活跃的仓库规模: llama.cpp 超 11.7 万 stars,Ollama 超 17.4 万 stars,vLLM 超 8.3 万 stars。

这一阶段的隐含假设是请求相对独立,服务端主要优化 tokens/s、TTFT、显存利用率。TGI 曾是代表性组件,但 HF 在 2025-12-11 将其置于维护模式,并推荐 vLLM/SGLang,说明推理引擎迭代速度已经超过第一代平台的维护边界。

当下: 2026 年的主战场是“agentic inference”

现在的工作负载不再是单轮问答,而是 agent 持续读取仓库、调用工具、生成 PR、触发 CI、等待反馈后继续修改。GitHub agent PR 的公开报道、AgentPerf 的新指标、OpenAI Agents SDK 的工具输出与 guardrail 迭代,都指向同一件事: 生产系统要优化的是“有 SLO 的连续 agent 会话”,不是一条 prompt 的 benchmark。

因此,KV cache 与调度成为中心。LMCache 在真实 agent trace 上给出 TTFT/p95/完成请求数改善,SGLang 将 HiCache/Radix Cache 作为重点,vLLM 的 DeepSeek V4 roadmap 把 KV offload、prefill/decode、FP4/FP8、硬件后端列为工程任务。这些都说明上下文复用正在变成一等资源。

未来: agent OS、cache economy 与低精度 rack-scale 会合

未来 6-18 个月内,我更看重三条可验证趋势。第一,agent 工作负载会推动代码平台、CI、review、artifact、向量索引和权限系统重构。第二,KV cache、tool result、retrieval context 会被当成可治理资产,出现更明确的生命周期、命中率、成本归因和跨 engine 复用。第三,硬件软件协同会继续向低精度和 rack-scale 走,AgentPerf 这类指标会把“多少并发 agent 能稳定跑”纳入容量规划。

风险也很明确: agent 生成速度超过验证层,会放大错误代码、供应链风险和无效 CI 消耗; MCP/server 工具生态如果缺少权限隔离和审计,会成为新的数据泄露面; 低精度推理若缺少模型级校验,可能在长链路推理中引入难定位的质量退化。

5. 趋势强度矩阵

6. 我的判断

  • 工程优先级: 如果只选一个方向跟进,优先跟 agentic inference: vLLM/SGLang 的调度与 cache、LMCache 的跨层复用、OpenAI/LangGraph 的 tool runtime、GitHub Actions 的 agent 工作流。
  • 成本优先级: 关注每个 agent session 的端到端成本,不要只看 tokens/s。指标应包括 TTFT、p95/p99、cache hit、CI minutes、tool call latency、失败重试次数、review backlog。
  • 架构优先级: 把 MCP gateway、tool 权限、审计、secret 隔离、输出 guardrail 设计成基础设施,不要散落在业务代码里。
  • 硬件优先级: 低精度和 rack-scale 的价值会首先体现在 MoE 与高并发 agent,但需要以模型质量回归和长链路任务通过率验证。
  • 迁移优先级: TGI 的维护模式提醒我们,serving engine 要留可迁移层: OpenAI-compatible API、模型配置抽象、benchmark harness、灰度切流和回滚机制。
保守假设: 当前公开数字来自不同来源,口径不完全一致。本文把它们作为趋势证据而不是精确审计结果。更可靠的做法是持续记录同一批 GitHub API 指标、release 频率、benchmark 口径和自身服务成本,形成自己的时间序列。
