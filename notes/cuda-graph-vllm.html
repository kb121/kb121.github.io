CUDA Graph 详解 & vLLM 里的 CUDA Graph 与精度

这篇笔记用图文 + 动图讲清三件事:① CUDA Graph 到底是什么、为什么能加速;② vLLM 里怎么用 CUDA Graph(piecewise / full);③ vLLM 里 CUDA Graph 模式 vs Eager 模式对精度的影响。基于 vLLM V1 引擎。

一句话总览: CUDA Graph = 把一连串 GPU kernel 录制成一张图,之后一次发射、整图重放,省掉每个 kernel 的 CPU 发射开销。它对 decode 阶段(大量小 kernel、访存密集)提速最明显。vLLM 默认用 FULL_AND_PIECEWISE(decode 用整图、prefill/混合用分段图)。CUDA Graph 本身不改数值,真正带来微小精度差异的是与之配套的 torch.compile(算子融合)和 batch padding。

0 · 先看一眼"问题"

GPU 跑一个 kernel,需要 CPU 先"发射(launch)"它 —— 这一步有固定开销(几微秒)。LLM 的一个 decode step 要串行跑成百上千个小 kernel(每层的 norm、QKV 投影、MLP、残差…)。当 batch 很小,单个 kernel 在 GPU 上跑得比 CPU 发射下一个还快,GPU 就只能干等 —— 这叫 CPU launch-bound。下面这张动图就是这个浪费:

1 · CUDA Graph 是什么

CUDA Graph 的思路非常直接:既然每次 decode step 跑的 kernel 序列几乎一模一样,那就把它录制(capture)成一张静态图,之后每步只发射这一张图、让 GPU 一口气重放(replay)所有 kernel,中间不再回到 CPU。

为什么对 LLM decode 特别有效? Decode 每步只生成 1 个 token,是 访存密集 + 小 batch:kernel 又多又小、发射开销占比高。CUDA Graph 正好把这部分开销抹平。反观 prefill:一次算很多 token,kernel 本身就跑得久,发射开销占比小,收益没那么大。

2 · CUDA Graph 的代价与约束

收益

  • 消除每个 kernel 的 CPU 发射开销,decode 延迟显著下降。
  • GPU 利用率拉满,小 batch 场景尤其明显。

约束

  • 静态形状:图固化了张量形状和显存地址,batch / 序列长度一变就得换一张图。
  • 无动态控制流:图里跳过所有 CPU 逻辑(if / print / Python 分支),只重放 GPU 算子。
  • 额外显存 + 启动耗时:每张捕获的图都要占显存,启动时 warmup 也要时间。

3 · vLLM 里的 CUDA Graph

LLM 模型里,注意力用的是 FlashAttention 这类专用 kernel,形状多变、对整图捕获不友好;而 非注意力部分(LayerNorm、MLP、残差、logits 投影)规整、好捕获。vLLM 据此提供了 分段(piecewise) 和 整图(full) 两种思路,并建立在 torch.compile 的分段编译之上。

vLLM V1 用一个 cudagraph_mode 开关统一管理这套策略,共 5 种模式:

cudagraph_mode含义适用
NONE完全关闭 CUDA Graph(= eager)调试 / 显存实验
PIECEWISE只用分段图,注意力等不兼容算子留 eager通用、最灵活
FULL整图捕获小模型 / 短 prompt
FULL_DECODE_ONLY只对 decode 批用整图,混合批不用图P/D 分离里的 decode 实例,省显存
FULL_AND_PIECEWISE 默认decode 批用整图,prefill / 混合批用分段图大多数模型最优
默认 FULL_AND_PIECEWISE 的巧思:decode 阶段对延迟最敏感 → 用整图把发射开销压到最低(降 ITL,即 token 间延迟);prefill / 混合阶段形状多变 → 用分段图保住灵活性和较小的 TTFT。两全其美。

4 · 形状离散化与 padding

CUDA Graph 要静态形状,但运行时 batch 大小千变万化。vLLM 的做法:只为一组预设的 batch size 捕获图(cudagraph_capture_sizes,如 1、2、4、8…),运行时把实际 batch 向上 padding 到最近的捕获尺寸,多出来的是 dummy 行,算完丢弃。

5 · CUDA Graph vs Eager:对精度的影响

这是最容易被误解的一节。先把结论摆出来,再拆开讲:

结论:CUDA Graph 本身不改变数值 —— 它重放的是和 eager 完全相同的 kernel,输入相同,输出逐比特相同。vLLM 里看到的 "graph 模式 vs eager 模式" 的细微差异,主要来自和 graph 配套的 torch.compile(算子融合、中间精度、kernel 选择),以及 padding 是否被正确 mask。

不会改数值的部分

纯 CUDA Graph 的"录制 → 重放":同样的 kernel、同样的顺序、同样的地址。重放 = 把 eager 那次执行原样再放一遍,数值逐比特一致。

可能带来差异的部分

  • torch.compile 融合:把多个算子融成一个 kernel,中间结果的累加顺序 / 精度变了 → 末位级差异。
  • kernel 选择:编译后可能选了不同实现(不同 tiling / 算法)。
  • padding 处理:dummy 行若没 mask 干净,可能影响 reduce / 归一化。
差异有多大? 通常是浮点末位级(在 fp16/bf16 容差内),极少改变最终采样出的 token;贪心解码下偶尔可能在某些 step 选到不同 token(累积效应),采样解码下基本是噪声级。PyTorch 官方口径:eager 与 compile 之间存在小数值差异是预期之内的,判断对错应与 fp64 参考比较,而非要求和 fp16 eager 逐比特相等。
实战调试法:怀疑精度问题(输出变差、复现某个 bug)时,第一步加 --enforce-eager 关掉 CUDA Graph + 编译,拿到"参考数值"做 A/B:
  • 加上后问题消失 → 差异来自 compile/graph 路径,多半是融合或 padding mask,可接受或针对性修。
  • 加上后问题还在 → 跟 CUDA Graph 无关,去查模型 / 权重 / 量化 / 输入。

6 · 实操速查

# 默认就是 FULL_AND_PIECEWISE,通常无需手动设
vllm serve Qwen/Qwen2.5-7B-Instruct

# 显式指定 cudagraph 模式
vllm serve MODEL --compilation-config '{"cudagraph_mode": "FULL_AND_PIECEWISE"}'
# 可选值: NONE | PIECEWISE | FULL | FULL_DECODE_ONLY | FULL_AND_PIECEWISE

# 纯 eager:关闭 CUDA Graph + torch.compile(调试 / 显存实验 / 拿参考数值)
vllm serve MODEL --enforce-eager

# 自定义捕获的 batch 尺寸(影响显存与命中率)
vllm serve MODEL --compilation-config '{"cudagraph_capture_sizes": [1,2,4,8,16,32]}'
什么时候用 --enforce-eager? ① 调试数值/精度;② 显存非常紧张,不想为捕获图额外占显存;③ 用的 attention backend 不支持图捕获;④ 频繁变化的奇异形状导致图反复 miss。其余情况保持默认(开图)就好,性能更优。
