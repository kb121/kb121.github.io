vLLM 完整调用链路拆解

这篇笔记把 vLLM 从 vllm serve 拉起服务、到一条 curl 请求打进来、再到模型前向、采样、流式返回的 完整调用链路 串成一条线,配多张图,帮你理清"vLLM 是怎么跑起来的、为什么快"。基于 V1 引擎(2025 起的默认架构)。

一句话总览: vLLM = OpenAI 兼容 API 服务 + 把 CPU 活(分词/调度/流式)和 GPU 活(前向)拆进不同进程 + 连续批处理(continuous batching) + PagedAttention 像操作系统分页一样管理 KV cache。这四点撑起了它"高吞吐"的核心卖点。

0 · 为什么需要 vLLM

朴素的 LLM 推理有两个致命浪费,vLLM 就是冲着这两点去的:

朴素实现的痛点

  • 显存碎片:给每个请求预留 max_len 的连续 KV cache,实际只用一点点,利用率常常 < 40%。
  • 静态批处理:一个 batch 里要等最慢的请求生成完才能整批结束,GPU 大量空转。

vLLM 的解法

  • PagedAttention:KV cache 切成固定大小的 block,按需分配、可共享,显存利用率拉到 > 90%。
  • 连续批处理:每一步(每次前向)都能让完成的请求退出、新请求插入,GPU 始终满载。

结果:相比朴素 HF Transformers 实现,吞吐量可达 10–24× 量级的提升(取决于模型与负载)。

想读源码时,先认清这几个最关键的目录,就能快速定位:

│   ├── llm.py            # 离线批量推理入口 LLM(...)

│   │   └── kv_cache_*    # KV cache 管理(block 分配)
│   ├── executor/         # Executor:管理多 GPU worker

│   │   └── gpu_model_runner.py  # ★ ModelRunner:装张量、跑前向

3 · 服务拉起:vllm serve 发生了什么

当你敲下 vllm serve Qwen/Qwen2.5-7B-Instruct,这一串事情按顺序发生:

  1. 解析参数,起 HTTP 服务
    CLI 解析 EngineArgs(模型名、TP 大小、gpu-memory-utilization 等),用 FastAPI + uvicorn 在 :8000 起 OpenAI 兼容服务。
  2. 构建 AsyncLLM,拉起 EngineCore 子进程
    AsyncLLM 通过 AsyncMPClient fork/spawn 出独立的 EngineCore 进程,两边用 ZeroMQ 做 IPC。
  3. Executor 起 Worker,每张 GPU 一个
    单机多卡用 MultiProcExecutor,每个 Worker 绑一张 GPU,内含一个 GPUModelRunner。
  4. 加载模型权重
    ModelRunner 调 model_loader 把权重读到显存(支持 safetensors / GGUF / bitsandbytes 等),按 TP/PP 切分到各卡。
  5. 显存 profiling + 分配 KV cache
    跑一次假的前向探测峰值显存,用 剩余显存 × gpu_memory_utilization 推算能放多少个 KV block,一次性把这块"显存池"占下来。
  6. 捕获 CUDA Graph(可选但默认开)
    对常见 batch 形状提前 capture CUDA Graph,消除 decode 阶段的 kernel launch 开销。
  7. 就绪,EngineCore 进入 busy loop
    服务打印 Application startup complete,开始空转等请求。
高频考点 —— gpu-memory-utilization:它不是限制权重大小,而是决定"显存里留多大一块给 KV cache 池"。调大 → 能容纳更多并发/更长上下文,但留给临时激活的余量变小,过大可能 OOM。默认 0.9。

4 · 模型 & KV cache "拉起"的本质

很多人以为"模型拉起 = 加载权重"。在 vLLM 语境里它其实是两件事,第二件才是重点:

① 加载权重

把 N B 参数读进显存,按张量并行(TP)切到各卡。这部分和别的框架差不多。

② 圈定 KV cache 池

权重之外的所有剩余显存切成等大 block 预先占住。这是 vLLM 高吞吐的物理基础。

6 · 连续批处理(Continuous Batching)

传统静态批处理:一批请求必须同生共死。连续批处理:以 step(一次前向)为粒度动态进出。

7 · PagedAttention & KV cache:像操作系统分页

这是 vLLM 的招牌,几乎必考。核心类比一句话:KV cache 之于 vLLM,就像物理内存之于操作系统。

背景:自回归生成里,每个已生成 token 的 Key/Value 要缓存下来给后续 token 做注意力(避免重算)。这块缓存就是 KV cache,它随序列变长而线性增长,是长序列推理的显存大头。

PagedAttention 把每个序列的 KV cache 切成固定大小的 block(比如每 block 存 16 个 token 的 KV),逻辑上连续、物理上可以散落 —— 就像 OS 的虚拟内存分页:

这种设计直接带来三个好处:

  • 几乎零碎片:按 block 粒度分配,不再为每个请求预留 max_len 连续空间,显存利用率 > 90%。
  • KV 共享(prefix caching):多个请求的相同前缀(如同一个 system prompt)可以指向同一批物理块,省显存又省重复 prefill。
  • copy-on-write:beam search / 并行采样分叉时,共享块只在写时才复制。

8 · 调度器:prefill 与 decode 的统一

V1 调度器做了个漂亮的简化:不再区分 prefill 和 decode 阶段,统一表示成"这一步给每个请求跑几个 token"。

Prefill 阶段

处理输入 prompt,一次算很多 token,算力密集(compute-bound)。延迟体现为 首 token 时间 TTFT。

Decode 阶段

每步只生成 1 token,要反复读全部 KV cache,访存密集(memory-bound)。延迟体现为 每 token 时间 TPOT。

Chunked Prefill:长 prompt 的 prefill 会一次性吃满 GPU、把正在 decode 的请求"饿死"造成卡顿。解法是把长 prefill 切成小块,和 decode 请求混在同一个 batch 里跑,兼顾吞吐与延迟。这也是统一调度表示带来的红利。

9 · 为什么拆成多进程(V1 的核心升级)

好处:GPU 进程永远不会因为"主线程在忙着分词/拼 JSON"而空等。CPU 活和 GPU 活真正并行,这是 V1 把吞吐进一步拉高的关键。代价是引入了 IPC 复杂度,V1 用"只传增量 + worker 缓存状态"把这个开销压到很低。

10 · 名词速查表

名词一句话解释
AsyncLLMCPU 侧异步包装层,管分词/反分词、对接 API、流式
EngineCore独立进程里的 busy loop,调度 + 执行一个 step
Scheduler维护 waiting/running 队列,按 token 预算决定每步跑什么
Executor / Worker每张 GPU 一个 worker,内含 ModelRunner
ModelRunner装配输入张量、跑前向、用 CUDA Graph 加速
PagedAttention把 KV cache 切成 block 按需分配/共享的注意力实现
KV block固定大小的 KV cache 单元(如 16 token),分配/回收的最小粒度
Continuous batching以 step 为粒度动态进出请求,GPU 不空转
Chunked prefill把长 prompt 的 prefill 切块,与 decode 混批,降卡顿
TTFT / TPOT首 token 时间 / 每 token 时间,两个核心延迟指标

11 · 一图流总结

一句话总结:"两进程(CPU/GPU 拆开)、一循环(EngineCore busy loop)、两优化(PagedAttention + 连续批处理)、一调度(token 预算统一 prefill/decode)。"
