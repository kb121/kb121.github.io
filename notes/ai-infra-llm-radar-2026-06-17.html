# 热点 统计/证据 为什么重要
1 GitHub 被 AI coding agents 改写容量曲线 DevOps.com 报道 GitHub 从 10x 容量计划升级到 30x; Agentic Brew 汇总早 2026 每周约 275M commits、Actions 2.1B minutes。 agent 把“开发者行为”机器化,让 Git、Actions、搜索、通知、权限、webhook、缓存、数据库一起承压。
2 vLLM v0.23.0: disaggregated serving 与 KV connector 成为主轴 6/15 release; GitHub API: 83,189 stars、18,157 forks、6/17 22:02Z push。release notes 明确列出 KV connectors、LMCacheMPConnector、NIXL、Mooncake、distributed serving 等。 vLLM 正从单机/单集群 serving engine 变成可插拔 KV transfer、prefill/decode 解耦和外部缓存生态的核心执行层。
3 SGLang v0.5.13: Radix/HiCache 与复杂生成程序 6/13 release; GitHub API: 29,137 stars、6,582 forks、6/17 22:02Z push。release notes 包含 HiCache/Radix Cache、DeepSeek V4、speculative decoding 等。 agent 工作负载有高 prefix overlap 和结构化输出需求,SGLang 的 RadixAttention/HiCache 方向贴近多轮、多工具场景。
4 LMCache v0.4.7: KV cache 独立成“数据层” 6/13 release; GitHub API: 9,261 stars、1,342 forks、6/17 21:14Z push。项目定位是最快 KV cache layer,服务 TTFT、throughput、跨 engine 复用。 长上下文、RAG、代码 agent 的重复 prefill 成本很高;KV cache 从临时显存状态变成可持久化、可观测、可调度资源。
5 Semantic KV cache reuse: 从精确 prefix 到语义复用 vLLM Semantic KV RFC 于 6/1 创建; SemBlend 描述面向 vLLM/SGLang 的 semantic donor discovery,GitHub API 显示 6/17 仍有 push。 如果语义相近但 token prefix 不同的请求也能复用部分 KV,长上下文推理经济性会从“缓存命中率”进入“检索/近似/质量控制”问题。
6 TensorRT-LLM: FP8/FP4、KV cache 与硬件特化 NVIDIA TensorRT-LLM 官方页列出 FP8/NVFP4 quantization、disaggregated serving、wide expert parallelism、speculative decoding; GitHub API: 13,899 stars、6/17 push。 当模型 API 能力趋同,生产差异来自硬件软件协同:低精度、kernel、KV cache dtype、speculative decoding 和 GPU 利用率。
7 Ollama + llama.cpp: 本地/边缘 runtime 继续扩张 GitHub API: ollama/ollama 174,397 stars、6/17 push; ggml-org/llama.cpp 117,035 stars、6/17 push。 隐私、离线、低延迟和成本控制推动“云端 frontier + 本地小模型/工具模型”的混合部署。
8 MCP servers 与 langchain-mcp-adapters: 工具协议工程化 GitHub API: MCP servers 87,387 stars; langchain-mcp-adapters 0.3.0 于 6/10 发布。 agent 的核心差异不只是模型,而是可连接哪些工具、如何鉴权、如何审计、如何隔离失败、如何在框架间复用。
9 LangGraph / OpenAI Agents SDK: agent runtime 走向状态机与可观测 GitHub API: LangGraph 35,054 stars、6/17 push; OpenAI Agents SDK v0.17.5 于 6/11 发布。 可靠 agent 需要 durable execution、state graph、trace、human-in-the-loop、sandbox 和工具预算,不是简单循环调用模型。
10 Hugging Face TGI 维护/归档: serving 代际切换完成 HF TGI docs 标注 maintenance mode; GitHub API 显示 text-generation-inference archived、10,863 stars。 第一代 LLM serving 的价值沉淀下来,新需求转向更活跃的 vLLM/SGLang/TensorRT-LLM/llama.cpp 组合。

3. 技术趋势分析: 历史、当下、未来

3.1 历史: 从“能跑模型”到“服务模型”

2022-2024 的核心问题是把 Transformer 模型稳定部署成 API:TGI 提供生产 serving 体验,vLLM 用 PagedAttention 和 continuous batching 改写吞吐,llama.cpp/Ollama 把量化模型带到本地,TensorRT-LLM 把硬件特化拉满。这个阶段的指标是 tokens/sec、TTFT、显存占用、batch size、OpenAI-compatible endpoint。

3.2 当下: Agentic workload 让瓶颈迁移

现在的瓶颈不再只是一次 chat completion 的吞吐。Agent 会反复读代码、检索文档、调用工具、运行测试、写 PR、触发 CI。于是系统瓶颈沿着这条链路迁移:

  • 证据 1: vLLM v0.23.0 把 KV connectors、disaggregated serving、LMCacheMPConnector 和 KV transfer 修复放在 release notes 的关键更新里。
  • 证据 2: SGLang v0.5.13 继续围绕 HiCache/Radix Cache、speculative decoding 与新模型优化迭代。
  • 证据 3: LMCache v0.4.7 与 vLLM/SGLang/KServe/Dynamo 生态相连,说明 KV cache 正在外置化。
  • 证据 4: GitHub 30x 容量报道说明 agent 输出会把软件协作平台本身也变成高并发 AI Infra。

3.4 未来: AI Infra 会变成“分层 agent 操作系统”

未来 6-18 个月,大概率不是某个单点 engine 获胜,而是出现更清晰的分层:闭源 frontier model 处理复杂推理,开源/本地模型承接成本敏感任务;vLLM/SGLang/TensorRT-LLM/llama.cpp 按 workload 分工;KV cache 从显存状态变成跨请求、跨节点、跨 engine 的调度资源;MCP 变成 agent 连接外部世界的契约;治理层负责审计、权限、预算和可复现。

4. 一句话结论

2026 年 AI Infra 的关键词是 agentic pressure:agent 把推理成本、上下文复用、低精度硬件、工具协议、CI/Git 平台容量和合规审计同时推到前台。过去拼模型调用,现在拼系统闭环;未来拼的是能否把模型能力稳定、便宜、可审计地接入真实工作流。
