3 SGLang v0.5.13: 结构化生成、多模态与 speculative decoding 6/13 release 包含 Nemotron 3 Ultra、Command A+、多类 diffusion 模型,并将 Spec V2 设为默认 speculative-decoding path;API 快照: 29,322 stars。 当 workload 从单轮问答变成 RAG、多轮、结构化输出和多模态生成,Radix/cache/programming model 的价值上升。
4 LMCache: KV cache 从 engine 内部状态变成基础设施层 GitHub API: 9,394 stars、1,353 forks、6/19 nightly CUDA 12.9/13.0 wheels;LMCache 博客强调长上下文场景的 TTFT、throughput 与 SLO。 Agent/RAG 会重复送入大量相同上下文,真正昂贵的是 prefill 重算;KV cache 命中率会变成服务质量指标。
5 NVIDIA Dynamo: 数据中心级分布式推理编排 NVIDIA Dynamo 文档称其支持 vLLM、SGLang、TensorRT-LLM,并通过 disaggregated serving、KV-aware routing、memory management 扩展 GPU fleet;GitHub API: 7,300 stars。 单个 engine 解决不了超大模型、MoE、长上下文和高并发的全局调度,Dynamo 代表“推理控制平面”上移。
6 KServe v0.19.0: Kubernetes-native LLM serving 继续产品化 6/14 release;KServe 文档列出 LLMInferenceService、LocalModelCache、KV Cache Offloading、Envoy AI Gateway、KEDA metric-based autoscaling。 企业部署需要 CRD、灰度、网关、限流、autoscaling 和可观测性,这让 LLM serving 融入云原生控制面。

3. 技术趋势分析: 历史、当下、未来

3.1 历史: 从“能跑模型”到“服务模型”

2022-2024 的主线是把 Transformer 模型部署成 API:连续批处理、PagedAttention、streaming、OpenAI-compatible endpoint、量化、本地运行时。TGI、vLLM、TensorRT-LLM、llama.cpp、Ollama 分别从生产服务、高吞吐、硬件优化、commodity hardware、本地体验切入。这个阶段的核心指标是 tokens/sec、TTFT、显存占用、batch size、模型加载速度。

证据是代际切换已经发生:HF 文档将 TGI 标为 maintenance mode 并推荐 vLLM/SGLang;vLLM 2026-06 release 仍有 200 contributors;SGLang 在结构化生成和 speculative decoding 上快速迭代;Ollama/llama.cpp 的 stars 规模说明本地运行时不是边缘现象。

3.2 当下: Agentic workload 让瓶颈迁移

当前最重要的变化是 workload 形态改变。传统 chat completion 更像“单次请求”;agent 会反复读取上下文、检索、调用工具、运行代码、写文件、开 PR、触发 CI。于是瓶颈沿着链路迁移:上下文切片影响 prefill,重复上下文带来 KV cache 需求,decode 端依赖低精度 kernel,工具调用依赖 MCP/guardrails,平台端承受 PR/CI/API fan-out。

  • 证据 1: GitHub 官方把 agentic development 直接归因到 repository creation、pull request activity、API usage、automation 和 large-repository workloads 的快速增长。
  • 证据 2: Dynamo 官方说明 disaggregated serving、KV-aware routing、optimized memory management,并支持 vLLM/SGLang/TensorRT-LLM。
  • 证据 3: KServe 文档把 LLMInferenceService、Envoy AI Gateway、KV Cache Offloading、metric-based autoscaling 放入 GenAI serving 能力。
  • 证据 4: OpenAI Agents SDK v0.17.6 加入 tool input guardrails,说明 agent runtime 正在把“人审/策略”前置到工具调用路径。

3.3 未来: AI Infra 会变成“分层 agent 操作系统”

未来 6-18 个月,竞争点大概率不是某个单点 engine 胜出,而是出现更清晰的分层和更强的控制面。推理 engine 仍重要,但它会被 cache-aware router、KV store、model gateway、agent state machine、tool protocol、policy/audit layer 包起来。也就是说,企业真正购买或自建的不是“模型调用”,而是一套能稳定完成工作的 agent 操作层。

  1. 模型层: frontier model 负责复杂推理,开源/本地模型承接隐私、低延迟和成本敏感任务。
  2. 推理层: vLLM / SGLang / TensorRT-LLM / llama.cpp 各自服务高吞吐、结构化生成、NVIDIA 极致优化和本地边缘。
  3. 缓存层: KV cache 从 engine 内部状态变成跨请求、跨节点、跨 engine 的资源调度对象。
  4. 控制面: Dynamo、KServe、Envoy AI Gateway、KEDA 把 GPU fleet、网关、限流、自动扩缩和灰度纳入统一操作面。
  5. 协议/治理层: MCP、guardrails、approval、trace、prompt/response hash、PII policy 和审计日志会成为企业默认要求。

4. 一句话结论

2026 年的 AI Infra 关键词是 agentic pressure:agent 把推理成本、上下文复用、低精度硬件、分布式控制面、工具协议、开发平台容量和合规审计同时推到前台。过去拼模型调用,现在拼系统闭环;未来拼的是能否把模型能力稳定、便宜、可审计地接入真实工作流。
