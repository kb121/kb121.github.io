3. 技术趋势分析: 历史、当下、未来

3.1 历史: 从 engine 优化到 serving 架构

历史上 AI Infra 的第一阶段重点是让单个模型更快:continuous batching、PagedAttention、CUDA graph、quantization、tensor parallel。第二阶段开始关心服务架构:prefill 和 decode 的资源画像不同,长上下文让 KV cache 成为瓶颈,单纯增加 GPU 不能线性解决 TTFT、tail latency 和成本问题。

  • 证据 1: Revisiting Disaggregated Large Language Model Serving 总结了 disaggregated serving 的性能与能耗权衡,说明 prefill/decode 拆分不是简单优化,而是服务架构选择。
  • 证据 2: Disaggregated Inference: 18 Months Later 提到 vLLM、SGLang、Dynamo、llm-d 等主流框架已经建设 first-class disaggregated inference 能力。
  • 证据 3: vLLM 和 SGLang 在一个月内多次发布,说明开源 engine 仍在快速吸收生产反馈。

3.2 当下: Router + KV cache + Kubernetes 控制面

当下最明显的变化是“有状态推理”被带进路由层。普通 round-robin 对无状态 HTTP 服务足够,但 LLM replica 持有不同 prefix/KV 状态,路由错误会导致重复 prefill。GKE Inference Gateway、llm-d Endpoint Picker、Gateway API Inference Extension 都在把 KV cache 命中率、queue depth、LoRA affinity 纳入请求级调度。

  • 证据 1: GKE Inference Gateway docs 明确说明 llm-d Router/EPP 依据 KV cache utilization、queue length、prefix cache state、LoRA adapter affinity 做 per-request endpoint decisions。
  • 证据 2: Google Cloud blog 将 llm-d 放在 Kubernetes AI Infrastructure 语境中,并强调 LWS、wide expert parallelism、prefill/decode 独立扩缩容。
  • 证据 3: 6/23 当天 LMCache 发布 v0.5.0,说明 KV cache 层正在以独立项目速度迭代。

3.3 未来: Agent OS 与推理成本市场化

未来一年的关键不是“哪个模型 API 更强”,而是“哪个平台能把 agent workload 的不确定性压成可管理的系统变量”。长时运行 agent 会产生代码执行、文件读写、MCP 工具调用、浏览器/CI、持久化 session、失败恢复和审计需求。推理侧则需要把 token、cache、GPU、网络、队列、sandbox 权限都纳入可观测和可调度体系。

  • 趋势 1: 推理成本会更像实时调度市场:prefill GPU、decode GPU、KV cache tier、network path、batch slot 都需要按 SLO 动态分配。
  • 趋势 2: Agent runtime 会逼迫 infra 提供 durable execution:session resume、workspace snapshot、sandbox isolation、safe output、审计日志会成为生产默认项。
  • 趋势 3: 开源 engine 竞争会从单点吞吐转向“平台可组合性”:vLLM/SGLang/TensorRT-LLM 是否能干净对接 router、cache、scheduler、observability,比单 benchmark 更重要。
